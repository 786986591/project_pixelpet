# project_pixelpet 技术方案

## 技术目标
V1 技术方案需要同时满足以下目标：
- 支撑 `桌面 Widget 优先` 的核心体验
- 避免依赖高风险权限与重后台保活
- 让宠物状态在离线条件下仍能稳定推导
- 控制首版复杂度，优先验证玩法与留存

## 技术栈选择
V1 推荐采用 Android 原生方案，不建议使用 `Unity` 或 `Flutter` 作为主栈。

客户端建议技术栈：
- `Kotlin`
- `Jetpack Compose`：App 内页面构建
- `Jetpack Glance`：桌面 Widget
- `Room`：宠物状态、道具、行为日志等本地存储
- `DataStore`：轻配置、首次引导、偏好设置
- `WorkManager`：周期刷新、提醒调度、状态补偿计算
- `Hilt`：依赖注入
- `Kotlin Coroutines + Flow`：异步与状态流转
- `Timber / Logcat` 或等价方案：日志调试
- `Firebase Analytics + Crashlytics`：可选的埋点与崩溃监控

## 为什么选择原生 Android
- `Glance + Compose` 已是官方推荐的 Widget 构建方向，和 App 内页面技术风格一致
- 原生方案更贴近 Widget 生命周期和系统限制，适合处理桌面组件细节
- V1 以 Widget 为核心，跨平台价值低于实现稳定性
- 避开额外运行时和跨端兼容成本，便于快速验证产品假设

## Widget 技术约束
### 更新频率限制
- `AppWidgetProviderInfo.updatePeriodMillis` 不能用来做高频刷新，官方限制最多约 `30 分钟` 一次
- 若使用 `WorkManager` 做周期任务，最小周期通常仍为 `15 分钟`
- 因此不能把真实状态维护建立在“每分钟后台刷新一次”的假设上

### 生命周期不稳定
- Widget 会因为系统重绘、桌面切换、设备状态变化而被动刷新
- 不同品牌桌面启动器对尺寸、刷新时机、布局行为存在差异
- 需要接受“系统不会严格按预期时间唤醒”的事实

## 状态更新原则
V1 的核心原则是：`真实状态通过时间推导得到，而不是通过高频后台任务维持。`

推荐模型：
- 宠物真实状态 = `基础快照 + 时间差衰减计算`
- 每次用户点击 Widget、打开 App、系统重绘 Widget、Worker 被触发时，重新计算当前状态
- 持久化层只记录最近一次可信快照与关键时间点，不记录每分钟结果

这样设计的收益：
- 即使系统没有准时执行后台任务，状态也不会整体错乱
- 同一套状态逻辑既可以服务 Widget，也可以服务 App 内详情页
- 更适合离线环境与低功耗目标

## 数据存储策略
### Room
用于存储：
- 宠物核心状态
- 互动行为日志
- 道具或背包数据
- 成长阶段记录

### DataStore
用于存储：
- 首次引导完成状态
- 用户轻偏好设置
- 低复杂度配置项

## 调度策略
`WorkManager` 在 V1 中主要承担辅助角色，而不是状态真源：
- 定时触发提醒
- 在合适时机请求 Widget 更新
- 处理长时间未进入 App 时的补偿逻辑

不应依赖：
- 精确闹钟驱动宠物状态流转
- 长驻后台服务维持 Widget 动画或状态同步

## 依赖注入与状态流
- 使用 `Hilt` 管理 Repository、UseCase、Scheduler、Updater 等对象依赖
- 使用 `Coroutines + Flow` 组织异步计算与 UI 状态响应
- App 与 Widget 尽量共用同一套领域状态计算逻辑，避免两套规则分叉

## 技术风险与规避
### 风险 1：Widget 刷新频率受限
- 规避方式：采用时间推导状态，不依赖高频后台刷新

### 风险 2：多机型桌面兼容差异
- 规避方式：首版只支持 `2x2` 与 `4x2` 两种尺寸，减少布局分支

### 风险 3：用户期待宠物像传统掌机那样持续运动
- 规避方式：把反馈重点放在点击响应、状态变化、关键帧切换，而不是持续动画

### 风险 4：若转向悬浮桌宠会引入高风险权限
- `SYSTEM_ALERT_WINDOW` 与 Overlay 方案有更高上架和用户信任风险
- V1 不应为追求“常动”效果而切换到该路线

### 风险 5：借助 AccessibilityService 取巧存在审核问题
- Google Play 对 AccessibilityService 有专门声明与审核要求
- V1 不应采用这类实现路径

## 当前技术结论
- V1 技术可行性判断为 `高`
- 核心实现难点集中在 `状态系统设计` 和 `Widget 刷新策略`
- 这两个问题在现有原生 Android 技术栈内可控，不需要引入更重框架
